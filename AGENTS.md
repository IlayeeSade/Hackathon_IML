# Agent guide for this repo

Hackathon image-classification project: classify images into 20 ImageNet
classes (see `labels.py` for the HF-index ↔ local-index ↔ name mapping).
Each team submits a `submissions/<team_name>/` folder; a harness loads and
scores every submission against a held-out validation set.

## Layout

```
labels.py            HF_INDEX_TO_NAME / HF_INDEX_TO_IDX / TARGET_HF_INDICES — the 20-class mapping
base_model.py         BaseModel ABC (load/predict contract) + reference ImageNetSubset Dataset
check_submission.py   Validates a submissions/<team>/ folder has the required files
evaluate.py           Older/unmaintained grading script at the project root — DATA_ROOT/SUBMISSIONS_DIR
                       are bare relative Paths and it expects a "validation" split that doesn't exist.
                       Don't use it as a reference; submissions/my_team/self_evaluation.py is the
                       fixed, working version (GPU-accelerated, points at dataset/val_split).

dataset/
  train_original/     Full source images per class (1000/class), the only folder with real data
  train_split/        Symlinks into train_original — regenerated by split_dataset.py
  val_split/          Symlinks into train_original — regenerated by split_dataset.py
  split_dataset.py    Rebuilds train_split/val_split from train_original (default 20% train / 80% val,
                       per-class, seeded shuffle). Re-run after editing ratios/seed.
                       --copy duplicates files instead of symlinking.

submissions/
  <team>/
    train.py          Trains ModelArchitecture, writes weights.joblib
    model.py           defines ModelArchitecture(nn.Module)
    predict.py          defines Model(BaseModel) — grader-facing wrapper; "students usually
                        should not change this file"
    weights.joblib     torch state_dict, loaded via joblib (not torch.save/load)
  my_team/             our submission — see below
  dummy_baseline/      reference baseline (predicts uniformly, ~5% acc on 20 classes)
```

## submissions/my_team/

- `model.py`: ResNet-50 (`torchvision.models.resnet50(weights=None)`), fc replaced with
  `Dropout(0.3) -> Linear(in_features, 20)`.
- `augmentations.py`: all the `transforms.Compose` builders (train/eval/stress-test variants).
- `train.py` flags:
  - `--resume`: load `weights.joblib` before training instead of starting fresh. Evaluates the
    loaded checkpoint's val metric first so a fresh epoch only overwrites it if it actually improves
    (otherwise a regressing first epoch would clobber a good resumed checkpoint).
  - `--save-best {acc,loss}` (default `acc`): which validation metric decides which epoch's weights
    get saved to `weights.joblib`.
- `self_evaluation.py`: runs every `submissions/*` team against 5 stress-test suites (clean, geometric,
  color, noise, "ultimate combo") on `dataset/val_split`, GPU-accelerated, tqdm progress bars with ETA.
  **Known bug**: the model used for the final visualization/confusion-matrix is whichever team
  evaluates first without crashing (alphabetical `team_dirs` order), not the best-scoring team — check
  the leaderboard printout for real rankings, not the "Generating ultimate visual stress-test for ..."
  line.

## Hardware/runtime gotchas (Vast.ai container)

- **`os.cpu_count()` lies.** It reports the host's full core count (128 here), but the actual cgroup
  CPU quota (`cat /sys/fs/cgroup/cpu.max`) is far lower (~13.65 cores on this box). Don't size
  `NUM_WORKERS` or `torch.set_num_threads()` off `os.cpu_count()` without sanity-checking the quota —
  oversubscribing causes severe throttling and was the proximate cause of an earlier "16 workers"
  training crash (CPU starvation → stalled workers → memory backup against the cgroup memory cap).
- **`free -h` also lies** about what this container can use — check `cat /sys/fs/cgroup/memory.max`
  for the real ceiling (~40GiB here vs. 125GiB host total).
- **GPU is a 12GB card (RTX 4070 Ti, ~11.72GiB usable).** ResNet-50 at full fp32 with a large batch
  size OOMs easily — activation memory scales ~linearly with batch size (~44MB/image empirically).
  `train.py` uses `torch.autocast` + `GradScaler` (AMP) specifically to afford a larger batch size;
  current `BATCH_SIZE = 192` was chosen empirically to leave real headroom (~2.9GiB) under that limit,
  not pushed to the theoretical max. Re-probe with a binary search over batch size (build model +
  optimizer, run 2 real fwd/bwd/step iterations, read `torch.cuda.max_memory_allocated()`) before
  raising it, rather than guessing.
- **Don't touch `dataset/train_split` or `dataset/val_split` while a training/eval run is active.**
  They're regenerated via `rm -rf` + symlink rebuild in `split_dataset.py`; doing this while a
  DataLoader is iterating them mid-run causes worker `FileNotFoundError`s and kills the run. `train.py`
  has no per-epoch checkpointing — only the final/best epoch is saved at the very end — so a crash
  like this loses the whole run's progress. Check `ps aux | grep train.py` (and check teammates'
  tmux panes) before touching the dataset splits.
- Resuming/parallel runs: avoid running multiple training or self-evaluation processes at once on this
  box — each Python process defaults to using many threads (`torch.get_num_threads()`), and two
  CPU-heavy jobs together will blow past the ~13.65-core quota and slow both down rather than truly
  parallelizing.
